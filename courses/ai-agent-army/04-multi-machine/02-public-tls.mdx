# 公网部署 — Caddy 反代 + TLS + 改密加固

> 公网部署不是把 `:9200` 放到互联网，而是把 Hub 留在本机回环地址，用 Caddy 接住 HTTPS，再把加密后的请求反代给 `127.0.0.1:9200`。

## 0. 一句话先讲明白

**生产公网入口只开 `443`，Caddy 负责 TLS 和反向代理，Hub 仍然跑在 `127.0.0.1:9200`；启动 Hub 和 Dashboard 后，立刻用 `anet login --username admin --password anethub` 登录，再用 `anet passwd` 改掉默认密码。**

上一节的 `anet hub start --host 0.0.0.0` 是局域网方案。公网方案反过来：Hub 不直接对公网监听，公网流量先到 Caddy，Caddy 用 HTTPS 终止 TLS，再转发到本机 Hub。这样不同城市的 Agent 可以通过你的 HTTPS Hub 地址加入同一个 Hub，而不是直接访问裸 HTTP 端口。

---

## 你将学到

- 为什么公网不要直接开放 `9200`
- Caddy 反代到 `127.0.0.1:9200` 的最小配置
- Hub、Dashboard、改密的正确顺序
- 不同城市的 Agent 如何用 HTTPS Hub 地址登录并启动节点
- 生产环境应该只开放哪些端口
- 上线前后要检查哪些安全项

## 适合谁

已经能在本机或局域网跑 anet，准备把多台不在同一局域网的机器接入同一个 Hub 的工程师。你需要会基本 Linux 命令、会编辑配置文件、知道云服务器安全组或防火墙在哪里配置。域名、证书、Caddy 安装方式因环境不同，本节给通用部署口径，不绑定某一家云厂商。

---

## 1. 公网部署的目标形态

推荐形态如下：

| 层 | 示例 | 对外开放 |
|---|---|---|
| Caddy | 你的域名 `:443` | 是 |
| CommHub Server | `127.0.0.1:9200` | 否 |
| Dashboard | `127.0.0.1:3000` 或受控入口 | 默认不公开 |
| Agent 机器 | 任意城市 | 主动访问 HTTPS Hub |

这套形态有两个关键点。

第一，Hub 只监听本机：

```bash
anet hub start
```

不要在公网机器上为了省事执行：

```bash
anet hub start --host 0.0.0.0
```

`--host 0.0.0.0` 在局域网章节里是为了让同网段机器连进来；在公网服务器上，它可能让所有网卡都接受 `:9200` 连接。公网入口应该由 Caddy 统一管理。

第二，外部 Agent 只认 HTTPS 地址：

```bash
read -r HUB_DOMAIN
HUB_URL="https://${HUB_DOMAIN}"
anet init --hub "$HUB_URL"
read -r MEMBER_USER
read -rsp MEMBER_PASSWORD
printf '\n'
anet login --username "$MEMBER_USER" --password "$MEMBER_PASSWORD"
```

这里的 `HUB_DOMAIN` 使用你控制的域名，并确保 DNS 指向运行 Caddy 的服务器。

## 2. 启动 Hub 和 Dashboard

在服务器上开第一个终端，启动 Hub：

```bash
anet hub start
```

默认情况下，`anet hub start` 绑定 `127.0.0.1`，端口是 `9200`。这个默认值正好符合公网反代的要求：Hub 只接受本机 Caddy 转发过来的请求。

如果需要观察节点和任务，可以开第二个终端启动 Dashboard：

```bash
anet hub dashboard
```

Dashboard 默认端口是 `3000`。生产环境不要默认把 `3000` 直接公开到公网。可以只在服务器本机使用、通过 SSH 隧道访问，或者给 Dashboard 单独配置受保护的 HTTPS 入口。对于本节最小生产形态，外部 Agent 并不需要访问 Dashboard，它们只需要访问 Hub API。

---

## 3. 先登录，再改掉默认密码

`anet hub start` 会 bootstrap 默认管理员账号。README 的 verified flow 写明默认账号是：

```text
admin / anethub
```

这组账号只适合第一次进入系统。公网部署时，正确顺序是：Hub 已启动，Dashboard 可选启动，然后 CLI 登录，再改密。

```bash
anet login --username admin --password anethub
```

登录成功后立刻改密：

```bash
anet passwd
```

`anet passwd` 是 CLI 命令参考里的已验证命令；源码里它调用 Hub 的 `/api/auth/password`，需要当前登录 token，并要求输入旧密码和新密码。也就是说，改密不是编辑本地配置文件，而是通过 Hub 的认证接口修改服务端账号密码。

如果你要在脚本化环境里操作，可以先阅读当前版本的 CLI 选项再决定是否传参。课程正文只推荐交互式 `anet passwd`，因为它不会把新密码留在 shell history 里。

改密后，旧的 `admin/anethub` 不应该再能登录。远端 Agent 不要长期复用 bootstrap 管理账号；本节先用低权限协作者账号示例接入，下一节再收口 network 角色和 token 边界。

## 4. Caddyfile：HTTPS 入口反代到本机 Hub

Caddy 的最小反代配置可以写成：

```caddyfile
{$HUB_DOMAIN} {
  reverse_proxy 127.0.0.1:9200
}
```

这段配置表达三件事：

1. 公网访问域名来自 `HUB_DOMAIN`
2. Caddy 负责处理 HTTPS 入口和证书
3. 请求被转发到本机的 `127.0.0.1:9200`

真实域名替换后，确保 DNS 已经指向这台服务器。Caddy 在常见部署下会自动申请和续期证书；如果你的环境使用内网证书、企业 CA 或手工证书，按你的基础设施规范配置 TLS，但反代目标仍然建议保持 `127.0.0.1:9200`。

启动或重载 Caddy 的命令取决于安装方式。使用 systemd 的 Linux 服务器常见流程是：

```bash
sudo caddy validate --config /etc/caddy/Caddyfile
sudo systemctl reload caddy
```

如果是第一次启动：

```bash
sudo systemctl start caddy
```

这些是 Caddy 的通用运维命令，不属于 anet CLI。anet 侧的事实只有一个：Hub 提供 HTTP 服务，默认在 `127.0.0.1:9200`；公网 TLS 和域名入口由反向代理负责。

配置完成后，在服务器本机检查后端：

```bash
curl http://127.0.0.1:9200/health
```

再从外部网络检查 HTTPS 入口：

```bash
curl "$HUB_URL/health"
```

第一条通、第二条不通，重点查 DNS、Caddy 配置、证书申请、防火墙和云安全组。第一条不通，先回到 `anet hub start` 终端看 Hub 是否运行。

## 5. 防火墙：公网只开 443，不直接开 9200

公网安全组或防火墙的目标状态应该是：

| 端口 | 是否公网开放 | 用途 |
|---|---:|---|
| `443/tcp` | 是 | HTTPS 入口，由 Caddy 接收 |
| `80/tcp` | 可选 | 证书签发、HTTP 到 HTTPS 跳转 |
| `9200/tcp` | 否 | Hub 后端，只给本机 Caddy |
| `3000/tcp` | 否 | Dashboard，默认不公网开放 |
| `22/tcp` | 受限 | 运维 SSH，只允许可信来源 |

有些证书自动签发流程需要 `80/tcp`，有些环境只用 DNS challenge 或已有证书则不需要。无论哪种方式，`9200/tcp` 都不应该直接对公网开放。

在服务器上，可以用本机 curl 验证后端存在：

```bash
curl http://127.0.0.1:9200/health
```

但不要把这个地址写给远端 Agent。远端 Agent 使用的是：

```bash
printf '%s\n' "$HUB_URL"
```

这能把认证、任务、SSE 等流量都放到 TLS 连接里，避免裸 HTTP 在公网传输。

## 6. 让不同城市的 Agent 加入同一个 Hub

在远端机器上安装 anet：

```bash
npm install -g @sleep2agi/agent-network
anet -v
```

配置 Hub 地址：

```bash
read -r HUB_DOMAIN
HUB_URL="https://${HUB_DOMAIN}"
anet init --hub "$HUB_URL"
```

登录。注意这里不要长期复用 `admin`。`admin` 只负责 bootstrap、改密和初始管理；远端 Agent 机器更适合用下一节讲的低权限用户或 member network 角色接入。下面用通用用户名表示：

```bash
read -r MEMBER_USER
read -rsp MEMBER_PASSWORD
printf '\n'
anet login --username "$MEMBER_USER" --password "$MEMBER_PASSWORD"
```

创建一个节点：

```bash
anet node create writer-a
```

启动它：

```bash
anet node start writer-a
```

第二台远端机器重复同样流程，只是换节点名：

```bash
read -r HUB_DOMAIN
HUB_URL="https://${HUB_DOMAIN}"
anet init --hub "$HUB_URL"
read -r MEMBER_USER
read -rsp MEMBER_PASSWORD
printf '\n'
anet login --username "$MEMBER_USER" --password "$MEMBER_PASSWORD"
anet node create reviewer-b
anet node start reviewer-b
```

远端 Agent 机器不需要开放入站端口。它们主动连到 Hub，接收任务推送并回传结果。对大多数个人服务器和小团队部署来说，这是比“每台机器都开公网端口”更容易维护的形态。账号、network 角色和 token 的最小权限策略放到下一节展开。

## 7. Dashboard 的公网策略

`anet hub dashboard` 是很有用的观察入口，但它不应该自动等同于公网管理后台。

推荐优先级如下：

1. 只在服务器本机使用 Dashboard
2. 用 SSH 隧道临时访问 Dashboard
3. 如确需公网访问，给 Dashboard 单独加 HTTPS、认证和来源限制

本节的 Caddyfile 只反代 Hub API：

```caddyfile
{$HUB_DOMAIN} {
  reverse_proxy 127.0.0.1:9200
}
```

不要为了方便直接开放：

```text
http://your-hub-domain:3000
```

Dashboard 能看到节点、任务和消息，属于控制面和观察面。它的安全级别不应低于 Hub。

## 8. 生产 checklist

上线前按这份清单逐项确认。

- Hub 使用 `anet hub start` 默认本机监听，没有用 `--host 0.0.0.0` 直接暴露公网
- Caddyfile 反代目标是 `127.0.0.1:9200`
- 公网入口使用 HTTPS，并保存在 `HUB_URL`
- 默认账号 `admin/anethub` 已经通过 `anet passwd` 改掉
- 远端 Agent 使用低权限协作者账号登录，不再使用默认密码
- 公网安全组只开放必要端口，至少不开放 `9200/tcp` 和 `3000/tcp`
- `curl http://127.0.0.1:9200/health` 在服务器本机可用
- `curl "$HUB_URL/health"` 在外部网络可用
- `anet init --hub "$HUB_URL"` 能在远端机器保存配置
- `anet login --username "$MEMBER_USER" --password "$MEMBER_PASSWORD"` 能在远端机器登录
- 每个节点使用无敏感信息的别名，例如 `writer-a`、`reviewer-b`
- 密码、token、API key 不写入公开文档、截图、仓库或 shell 分享记录
- Dashboard 没有裸露公网；如需公网访问，已单独加认证和来源限制
- 服务器日志、Caddy 日志、Hub 终端输出有基本留存，便于排查

这份 checklist 不是大型企业安全基线，但足够挡住最常见的错误：默认密码未改、HTTP 裸传、`9200` 直接暴露、Dashboard 随手公开。

## 9. 常见问题排查

如果远端机器 `anet init --hub "$HUB_URL"` 失败，先用 curl 看 health：

```bash
curl "$HUB_URL/health"
```

如果 HTTPS health 不通，在服务器本机确认 Hub：

```bash
curl http://127.0.0.1:9200/health
```

后端通、HTTPS 不通，查 Caddy、DNS、证书和防火墙。后端不通，查 `anet hub start` 是否还在运行。

如果 health 通但登录失败，检查是否已经改密。公网部署后，`admin/anethub` 应该只在第一次登录时使用；改密后继续拿默认密码登录会失败，这是正确现象。

如果登录成功但节点启动后不在线，运行：

```bash
anet status
anet doctor
```

`anet status` 看 Hub 视角里的节点和任务；`anet doctor` 做本机诊断。再检查节点机器的模型供应商 key、runtime 登录态和当前工作目录下 `.anet/nodes/writer-a/config.json` 是否是你刚创建的节点。

## 总结

公网部署的原则是端口收敛和默认拒绝：Hub 只在 `127.0.0.1:9200` 服务，Caddy 负责 `HUB_URL` 对应的 HTTPS 入口，公网只直接接触 `443`。不要把上一节的局域网命令 `anet hub start --host 0.0.0.0` 直接搬到公网服务器。

改密是上线动作，不是可选优化。启动 Hub 和 Dashboard 后，先 `anet login --username admin --password anethub`，再 `anet passwd`。远端 Agent 使用 HTTPS 地址和低权限协作者账号接入，账号与 network 角色的收口在下一节继续处理。

完成后，你就有了一个可以跨城市协作的 anet Hub：不同机器上的 Agent 主动连入同一个安全入口，Hub 统一记录任务、消息和在线状态。
